# 产品需求文档（PRD）

# 1. 文档信息
- 产品名称：OfferFlow
- 版本：V1.0（MVP + 增强版规划）
- 文档类型：Web端产品PRD
- 目标：围绕“岗位JD + 简历 +个人经历数据库 + JD相关数据库”，提供一体化的简历优化、模拟面试、投递管理闭环，帮助求职者提高投递命中率与面试通过率。

# 2. 产品背景与机会
# 2.1 用户痛点
-  简历与JD匹配低效：用户很难把自身经历映射成岗位真正关注的“核心特质”。
- 缺少可量化的简历质量评估：用户无法客观判断修改前后是否真的变好。
-  准备面试碎片化：面经、行为题、业务题分散，缺少连贯追问与复盘。
-  投递流程失控：多岗位并行时，记录混乱，无法复用历史简历版本和经验。
- 个体能力无法结构化呈现：没有实习时，无法有效转化项目/科研/志愿经历。
# 2.2 产品机会
- 将用户已有材料（简历历史版本、项目文档、代码、论文、PPT）沉淀为可检索“能力证据库”；
- 建立“JD解析 → 简历改写 → 模拟面试 → 投递跟进 → 真实面试复盘”闭环；
- 通过版本化和标签化，使每次求职沉淀可复用，越用越准。

# 3. 产品目标与成功标准
# 3.1 产品目标
- 提升简历与JD匹配质量，减少用户“从零改简历”的时间成本；
- 提升模拟面试真实性与针对性，帮助用户形成可执行优化路径；
- 构建可追踪的投递管理体系，覆盖求职全生命周期。
# 3.2 核心KPI
- 简历优化完成率（启动后生成可用版本）≥ 75%
- 用户采纳修改率（接受的建议条目/总建议条目）≥ 50%
- 模拟面试轮次完成率（单JD完成一面/二面/HR面各至少1次）≥ 33%
- 投递记录完整率（关键字段不缺失）≥ 80%
- 次周留存（有新增操作）≥ 30%

# 4. 目标用户与典型场景
# 4.1 用户画像
-  应届生/实习生：经历少，难以“包装”有效故事。
- 社招转岗用户：需要将过往经验翻译到新岗位语言。
- 高频投递用户：同时管理多岗位，需强流程工具。
# 4.2 典型场景
- 用户拿到一个新JD，上传简历，快速得到“岗位核心特质+逐条改写建议”。
- 用户在正式面试前按一面、二面、HR面进行模拟面试，并获得优先级改进清单。
- 用户投递后持续记录状态，并将真实面试内容反哺下一轮模拟与简历版本。

# 5. 版本规划
# 5.1 必做（MVP）
- 输入四件套：必传JD、必传简历（或简化简历表单）、可选个人经历数据库文件夹、可选JD相关数据库（如面经、岗位笔记、公司资料）
- 三大功能入口：简历优化、模拟面试、投递管理
- JD结构化解析（含实习/正职区分）
- 简历逐条建议+证据链+接受/拒绝操作+优化前后评分对比
- 模拟面试（按一面/二面/HR面轮次，含追问、评分、复盘）
- 投递看板与状态管理（半自动+手动字段）
# 5.2 增强（V1.5+）
- 多来源趋势参考（如公开平台面经/经验贴）
- 岗位相似度匹配与历史版本推荐
- 智能提醒（如“已7天未跟进”）
- 高级导出（不同模板投递版简历）
# 5.3 暂不做（V2后）
- 语音实时面试（先以文本模式为主）
- 企业内推社交网络模块
- 第三方招聘平台自动投递（受限于平台规则）

# 6. 信息架构与导航
- 顶部导航：工作台｜简历优化｜模拟面试｜投递管理｜资料库
- 工作台模块：
    - 最近JD任务
    - 本周待办（待投递/待模拟/待复盘）
    - 关键指标（投递数、面试率、通过率）
- 每个JD形成“任务空间（Job Workspace）”，三功能共享上下文与历史。

# 7. 核心功能详细需求
# 7.1 输入与初始化模块（统一入口）
# 7.1.1 输入项
1. JD（必传）
    - 支持：文本粘贴、文件上传、URL抓取（后续）
    - 校验：岗位名、职责、要求字段完整度检查
2. 简历（必传）
    - 支持：新文件上传、历史文件选取（格式：PDF/DOCX/Markdown）
    - 若无简历，进入“简化简历表单”
3. 个人经历数据库（可选）
    - 支持上传文件夹/多文件
    - 文件类型：PDF、DOCX、PPT、TXT、MD、代码文件等
    - 可包含：历史简历版本、项目报告、论文、志愿经历材料
4. JD相关数据库（可选）
    - 支持上传文件夹/多文件
    - 文件类型：PDF、DOCX、PPT、TXT、MD、图片转文本结果等
    - 可包含：该岗位面经、岗位研究笔记、公司业务分析、历史投递反馈
    - 用途：为简历优化、模拟面试提供外部补充参考
# 7.1.2 简化简历表单（无简历用户）
- 必填：基本信息、教育、至少1段经历（项目/科研/志愿任一）
- 选填：技能、奖项、证书、链接（GitHub/作品集）
- 生成“基础简历V0”，供后续优化使用
# 7.1.3 系统处理
- 文档解析 → 字段抽取 → 向量化索引（用于证据检索）
- 生成本次JD任务ID
- 自动进入三功能选择页
# 7.2 功能一：简历优化
# 7.2.1 目标
围绕目标JD，把用户经历“翻译”为岗位需要的能力证据，输出可投递的高匹配简历版本。
# 7.2.2 关键流程
1. JD深度解析
- 判断：实习 / 正职
- 输出：
    - 岗位画像（候选人类型）
    - 好简历标准（该岗位版本）
    - 5条核心特质（该岗位必须有,按优先级排序）
    - 关键词词云（技能、业务、软素质）
2. 候选人资产盘点
- 从简历+资料库抽取可用经历点
- 对应到5条核心特质，形成“覆盖矩阵”
3. 差距分析与补齐建议
- 已覆盖项、弱覆盖项、缺失项
- 输出未来2-4周可执行提升建议（按优先级）
4. 当前简历分析与打分（优化前）
- 评分维度（各0-20分，总分100）：
    - 竞争力（与目标JD的区分度与优势强度）
    - 内容完整性（教育、经历、技能、项目、关键信息完整度）
    - 专业性（术语准确、表述专业、岗位语言一致性）
    - 成果量化（是否有指标、结果、影响范围）
    - 结构规范性（排版结构、层次、可读性、时序一致性）
- 输出：总分、分项分、扣分依据、优先优化项
5. 逐条改写建议（仅改有增益项）
- 每条包含：
    - 原句
    - 建议改写
    - 支撑证据（来自JD条款 + 用户材料片段）
    - 改写目的（匹配哪条核心特质）
- 策略约束：
    - 对“已高质量且改动收益低”的内容标记为“保持不变”，不强行修改
    - 仅输出可同时提升“JD匹配度”或“简历评分”的改动项
- 用户可接受/拒绝/手改
6. 版本生成与沉淀
- 输出简历版本Vn
- 绑定“核心属性标签”（例：大厂-算法实习、后端开发-社招）
- 保存历史可回滚，可基于任一版本继续优化
7. 优化后复评打分
- 对优化后的版本按同一维度再次打分
- 输出“优化前后对比报告”：总分变化、分项变化、仍待提升项
# 7.2.3 特殊规则
- 若无实习经历，不惩罚：自动优先挖掘项目/科研/竞赛/志愿经历。
- 若资料冲突（时间线矛盾），提示用户确认。
- 同一JD可反复迭代，支持增量上传新资料。
# 7.2.4 输出物
- 《JD解析报告》
- 《匹配度诊断（含5核心特质评分）》
- 《简历评分报告（优化前/后对比）》
- 《逐条修改清单（可交互采纳）》
- 《优化后简历（可导出PDF/DOCX）》
# 7.3 功能二：模拟面试
# 7.3.1 目标
基于JD与用户背景，提供高拟真、可追问、可复盘的面试训练系统。

# 7.3.2 轮次与题目结构
- 轮次优先：先选择面试轮次，再按题型比例抽题
- 支持轮次：一面、二面、HR面
- 题型池：自我介绍、行为面、经历面、业务面、行业洞察
- 基线要求：每场均包含“自我介绍”
- 抽题策略（可配置）：
    - 一面：经历面/业务面为主，辅以少量行为面
    - 二面：业务深挖/项目难点/系统思考为主，追问强度更高
    - HR面：行为面/动机匹配/稳定性与沟通为主
- 行为面（协作、冲突、成长）
- 经历面（项目细节、难点、指标）
- 业务面（岗位相关业务理解）
- 行业洞察（趋势、竞品、方法论）
# 7.3.3 核心机制
1. 按轮次生成模拟场次（每JD）
    - 一面：至少2次，且尽量问题不同
    - 二面：至少2次，且尽量问题不同
    - HR面：至少2次，且尽量问题不同
    - 去重策略：题干语义相似度阈值控制，避免重复问同一问题
2. 动态追问
    - 基于用户回答实时生成追问（澄清、反例、量化、复盘）
3. 答后评估
- 每题给出：
    - 参考高质量答案
    - 考察点
    - 作答思路框架
    - 用户回答复盘（内容、结构、说服力、风险点）
4. 总评与行动建议
- 输出“问题清单 + 优先级优化路径”（例如先补STAR结构，再补业务深度）
# 7.3.4 真实面试反哺
- 用户可上传真实面试Q&A
- 系统提取：
    - 高频问题
    - 卡壳问题
    - 面试官关注点
- 自动用于后续模拟题调整，并对“类似岗位”形成经验迁移建议
- 《真实面试复盘报告》：
    - 逐题问题诊断（回答偏题、深度不足、量化缺失、结构混乱等）
    - 轮次维度诊断（在一面/二面/HR面分别的主要短板）
    - 后续提升方向（按优先级与可执行动作输出）
# 7.4 功能三：投递管理
# 7.4.1 目标
可视化管理每个JD的求职进度，打通“优化-面试-投递-复盘”。

# 7.4.2 数据字段（按JD记录）
- 基础：公司、岗位、城市、类型（实习/正职）、来源渠道
- 自动字段：
    - JD解析时间
    - 简历优化前评分、优化后评分、分项分变化
    - 简历优化次数
    - 模拟面试次数与场次记录
    - 是否上传真实面试复盘
- 手动字段：
    - 是否投递
    - 投递日期
    - 是否收到面试邀约
    - 面试轮次状态（1面/2面/HR面）
    - 流程终止节点与原因
# 7.4.3 分类与检索
- 按关键词（技术栈、方向、职级）分类
- 按进度看板（待优化/待投递/面试中/已结束）
- 按结果筛选（offer/拒信/无反馈）
# 7.4.4 关键页面
- 投递看板（Kanban）
- 岗位详情时间线（完整操作轨迹）
- 统计页（投递转化漏斗）

# 8. 智能能力设计（算法与规则层）
# 8.1 JD解析引擎
- 实习/正职分类器
- 核心特质抽取（Top5）
- 能力维度映射：技术能力、业务理解、执行力、协作力、成长性
# 8.1.1 多源资料融合
- 将JD原文、个人经历数据库、JD相关数据库进行统一检索与加权
- 参考优先级：JD原文 > 用户真实经历证据 > JD相关参考资料
- 目标：确保建议“有据可依”，同时不过度依赖外部泛化话术
# 8.2 证据检索与可解释性
- 从用户材料中检索支持证据
- 每条建议必须带“来源证据”，避免黑箱改写
# 8.3 评分体系（示例）
- JD匹配度：0-100
- 简历总分（是否好）：0-100（竞争力、内容完整性、专业性、成果量化、结构规范性）
- 简历可信度系数（是否真实）：0.0-1.0（证据可验证性、表述一致性、量化可信度；单独展示，不与简历总分重复计分）
- 最终建议排序分：简历总分 × 简历可信度系数（用于建议优先级与风险提示）
- 面试表现：题级分 + 维度分 + 总分
# 8.4 安全护栏
- 避免虚构经历：仅在用户材料证据范围内改写
- 对“夸大风险表达”给警示提示

# 9. 数据模型（核心实体）
- User
- JobJD（含解析结果、标签）
- ResumeBase（原始）
- ResumeVersion（V1、V2…，关联标签）
- EvidenceAsset（文件/片段/来源）
- OptimizationSession（一次优化过程）
- InterviewSession（一次模拟）
- RealInterviewRecord（真实面试上传）
- ApplicationRecord（投递状态）
实体关系：JobJD为主轴，关联简历版本、面试记录与投递状态，形成完整闭环。

# 10. 交互与体验要求（UX）
- 引导式流程：首次进入用3步向导完成初始化
- 低负担交互：逐条“接受/拒绝/编辑”而非整页重写
- 关键可见性：每条建议都能看到“为什么改”
- 历史可追踪：版本差异对比（Diff）+ 一键回滚
- 长流程防中断：自动保存草稿，支持续接

# 11. 非功能需求
- 性能：单次文档解析 < 20s（中等文件量）
- 稳定性：关键流程成功率 > 99%
- 安全：用户文件加密存储、传输HTTPS
- 隐私：默认不外泄用户材料；可一键删除项目空间
- 审计：关键操作日志可追溯（版本生成、状态变更）

# 12. 异常与边界场景
- JD内容过短：提示补充职责/要求后再分析
- 简历解析失败：提供手动字段补录
- 文件过大或格式不支持：分片上传/格式转换提示
- 信息冲突：时间线冲突检测并二次确认
- 用户拒绝全部建议：保留会话并推荐“轻改模式”

# 13. 埋点与分析方案（关键事件）
- jd_uploaded
- resume_uploaded_or_form_completed
- optimization_started/completed
- suggestion_accepted/rejected/edited
- interview_session_started/completed
- real_interview_uploaded
- application_status_changed
用于漏斗分析：上传 → 优化完成 → 模拟完成 → 投递 → 面试邀约。

# 14. 发布计划
# Phase 1（4-6周，MVP）
- 输入三件套 + 可选JD相关数据库
- JD解析 + 简历优化（逐条采纳）
- 模拟面试（按轮次抽题+复盘）
- 投递管理基础看板
# Phase 2（6-10周）
- 真实面试反哺机制
- 一面/二面/HR面多场次策略化模拟（每轮至少2次）
- 历史版本智能推荐（相似岗位复用）
# Phase 3（10周+）
- 高级统计与求职策略建议
- 模板市场与多格式精细导出
- 智能提醒与节奏管理

# 15. 验收标准（UAT）
- 用户可在无简历情况下，完成表单并生成可优化简历。
- 每个JD都可生成5条核心特质与对应匹配分析。
- 系统可对简历在优化前与优化后分别打分，并输出五维分项与总分变化。
- 简历优化建议不强制覆盖全简历，需支持“保持不变”并仅修改有明确增益项。
- 简历建议支持逐条采纳/拒绝，并能生成新版本。
- 系统支持上传JD相关数据库，并可在简历优化与模拟面试中被引用为参考来源。
- 每个JD在一面、二面、HR面均可完成至少2次模拟，且问题尽量不同，并有复盘报告。
- 投递管理可查看自动字段（优化次数、模拟次数）与手动字段（投递日期等）。
- 用户可上传真实面试记录，系统可输出真实面试复盘并影响后续模拟题策略。

# 16. 风险与应对
- 风险1：建议泛化、缺少个性化
    - 应对：强化证据检索与岗位标签，要求每条建议绑定来源。
- 风险2：用户材料质量低
    - 应对：内置“补充提问”机制，引导补齐量化信息。
- 风险3：用户对AI可信度不足
    - 应对：展示“改写理由+证据链+可回滚”。
- 风险4：流程长导致流失
    - 应对：任务拆分、进度条、自动保存、短闭环成果展示。

# 17. 总结
- 从“三个独立功能”升级为单JD任务空间闭环；
- 增加了“证据链+可解释改写+版本标签复用”；
- 明确“无实习经历”替代策略（项目/科研优先）；
- 将真实面试内容纳入训练反馈，形成持续学习系统；
- 投递管理不只是记录，而是连接优化与面试效果的数据中枢。
